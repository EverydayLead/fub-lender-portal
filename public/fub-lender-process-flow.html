FUB Lender — Process Flow & Business Requirements

Customer Journey × Banker Workflow — Operational Reference

1

Initiate Deal

Customer expresses need — banker opens deal in the field or at desk, system pre-fills from core

C Customer Experience
Expresses Financing Need
Contacts banker in person, by phone, or via referral. "I need to buy cattle / equipment / land / expand my shop."
Provides Basic Info
Name, amount, purpose, collateral type. If existing customer, banker already has this from core.
Feels Speed & Confidence
No long intake form. Banker captures the story on the spot. Customer sees professionalism, not paperwork.
B Banker Workflow
Open New Deal
Tap "New Deal" on mobile app. Search/select borrower from Jack Henry core.
System Auto-Pulls Entity Data
Legal name, TIN, address, existing deposit/loan relationships, prior financial statements — all pre-filled.
Confirm Deal Parameters
Amount, purpose, loan type, collateral class. System classifies deal into complexity lane.
Complexity Routing — System classifies deal at initiation
Fast Lane
Under $250K secured
Single borrower/guarantor
Standard collateral (auto, ag, equipment)
Target: <15 min initiation to submit
Standard
CRE or C&I deals
1–2 entities
Standard financial analysis
Full credit memo with spreads
Complex
Multi-entity / holding company
Participation or SBA
Construction / phased draws
Committee-level approval required
System
Jack Henry Core Integration (SilverLake / Symitar)
Real-time lookup of customer master, deposit accounts, existing loans, prior financials, and guarantor relationships. System writes back deal-level data post-booking for single-system-of-record integrity.
Operations Detail
  • Jack Henry API call must return within 3 seconds on LTE; offline mode caches last-known entity data
  • Duplicate borrower detection runs on TIN + legal name fuzzy match before deal creation
  • Complexity lane is system-suggested but banker can override with reason code (audit-logged)
  • Deal ID generated at creation — all downstream artifacts reference this ID
  • OFAC / BSA screening triggered automatically at deal open (non-blocking; flags if hit)
Business Requirements — Phase 1
Must Have (MVP) Should Have (v1) Could Have (v2)
BR-1.1 · Must
System shall retrieve borrower entity data from Jack Henry core in real-time at deal creation
Ops: API SLA <3s P95. Fallback to cached data if core unavailable. Alert ops if cache >24h stale.
BR-1.2 · Must
System shall auto-classify deals into Fast / Standard / Complex lanes based on amount, collateral type, and entity count
Ops: Classification rules configurable by loan administration without code deploy. Override audit log retained 7 years.
BR-1.3 · Must
Banker shall be able to initiate a deal from mobile device in the field with offline tolerance
Ops: Offline queue syncs on reconnect. Conflict resolution: last-write-wins with merge log. Max offline window: 72h.
BR-1.4 · Should
System shall run OFAC/BSA screening at deal initiation and flag hits without blocking workflow
Ops: BSA officer receives real-time alert on hit. False positive clearance workflow required. Screening vendor: existing FUB contract.
BR-1.5 · Should
System shall detect potential duplicate borrower entries and prompt banker to link to existing entity
Ops: Fuzzy match on TIN + name. Dedup rate target: <1% duplicate entities created.
BR-1.6 · Could
System shall pre-populate guarantor and related-entity relationships from core data
Ops: Requires Jack Henry relationship graph API. Verify accuracy with 50-deal sample before production.
2

Build Credit Memo

Banker writes the credit story — system extracts structured data. Not form-first.

⏱ Fast Lane target: <15 minutes from deal open to submit
C Customer Experience
Tells Their Story
Customer explains the deal narrative: what they're buying, why, how it fits their operation. Banker captures it naturally.
Provides Documents
Hands over tax returns, financials, appraisals — or banker photographs them on the spot. No portal login required at this stage.
Confirms Key Terms
Banker reviews proposed rate, term, and structure. Customer agrees or negotiates. No surprises later.
B Banker Workflow
Write Credit Narrative
Free-text credit story. System parses purpose, strengths, weaknesses, mitigants into structured fields. Banker reviews/corrects.
Capture Docs & Collateral
Photo/scan documents. Enter collateral descriptions and values. Attach or enter financial data. All async — nothing blocks forward progress.
Set Terms & Pricing
Rate, term, amortization, fees. System suggests based on product type and risk tier. Banker adjusts.
Financial Spreads (Standard / Complex)
Enter or import financial statements. System calculates DSCR, leverage, liquidity. Trend analysis auto-generated.
System
Progressive Disclosure Engine
Shows only lane-relevant fields. Fast Lane sees 8–10 fields. Standard sees full credit memo template. Complex adds participation, construction draw, and SBA-specific sections. System validates inline and flags missing required items with non-blocking indicators.
Operations Detail
  • Credit narrative NLP extraction is advisory — banker must confirm every parsed field before submit
  • Document attachments stored in encrypted blob storage; virus scan on upload; max 25MB per file, 200MB per deal
  • Financial spread calculations must match FUB's existing spreading methodology (validated against current Excel templates)
  • Auto-save every 30 seconds. Incomplete deals persist indefinitely in "Draft" status with 90-day stale alert
  • All field changes audit-logged with timestamp, user, old value, new value
  • Pricing engine floor/ceiling guardrails set by treasury; exceptions require approval tier escalation
Business Requirements — Phase 2
Must Have (MVP) Should Have (v1) Could Have (v2)
BR-2.1 · Must
System shall support narrative-first credit memo entry where banker writes the credit story and system extracts structured data
Ops: NLP extraction accuracy target: >85% on field mapping. Banker confirmation required before submit. Model retraining quarterly.
BR-2.2 · Must
System shall display only lane-relevant fields (progressive disclosure) based on deal complexity classification
Ops: Field rules configurable by loan admin. Changes require QA review of all three lane templates before deploy.
BR-2.3 · Must
Banker shall capture documents via camera, upload, or scan with OCR extraction
Ops: OCR for tax returns, financial statements. Extraction results must be human-reviewed. Retention per FUB document policy (7yr min).
BR-2.4 · Must
System shall calculate DSCR, leverage ratios, and liquidity from entered/imported financials
Ops: Calculations must match existing FUB spreading methodology. Parallel run against Excel templates for first 100 deals.
BR-2.5 · Should
System shall suggest pricing (rate, fees) based on product type, risk tier, and current rate environment
Ops: Pricing model updated by treasury weekly. Floor/ceiling overrides logged and routed to treasury review.
BR-2.6 · Should
System shall auto-save deal state and support resume from any device with conflict resolution
Ops: Save interval: 30s. Offline conflict resolution: last-write-wins with full diff log. 90-day stale draft alert to manager.
BR-2.7 · Could
System shall generate trend analysis charts from multi-year financial data
Ops: Requires 3+ years of financials. Charts embedded in credit memo PDF export.
3

Submit & Route

Completeness validated — deal routed to the right approver automatically

C Customer Experience
Told "We're Moving Forward"
Banker communicates that the deal is submitted for approval. Customer has clarity on next steps and expected timeline.
No Waiting in Limbo
Unlike current process, customer doesn't wait days wondering what's happening. Banker has real-time status to share proactively.
B Banker Workflow
Pre-Submit Completeness Check
System validates all required fields for the lane. Missing items shown as checklist — must-haves block submit, nice-to-haves are flagged.
Submit for Approval
One-tap submit. Deal locked for editing (lender can unlock with reason). Timestamp and snapshot captured.
Track Status in Real Time
Deal card shows: Submitted → In Review → Approved/Declined/Conditions. Push notification on decision.
System
Approval Authority Routing Engine
Maps deal amount, type, and risk characteristics to the approval authority matrix. Routes to individual approver (below committee threshold) or loan committee queue. Supports sequential and parallel approval chains. Escalation timer if no action within SLA window.
Operations Detail
  • Approval authority matrix maintained by Chief Credit Officer; updated at minimum annually
  • Routing rules: individual authority thresholds by officer, committee thresholds by deal type
  • SLA clock starts at submit. Default SLA: Fast Lane 4h, Standard 24h, Complex 72h (business hours)
  • Auto-escalation: if primary approver doesn't act within 50% of SLA, secondary approver notified
  • Deal snapshot at submit is immutable — any changes require unlock + re-submit + new snapshot
  • Notification channels: push notification (primary), email (secondary), SMS (escalation only)
Business Requirements — Phase 3
Must Have (MVP) Should Have (v1) Could Have (v2)
BR-3.1 · Must
System shall validate deal completeness against lane-specific required fields before allowing submit
Ops: Validation rules version-controlled. Changes require credit policy sign-off. Must-have vs. nice-to-have distinction per field.
BR-3.2 · Must
System shall route submitted deals to the correct approver based on the approval authority matrix
Ops: Matrix maintained in admin UI by CCO. Supports individual authority limits, committee thresholds, and delegation during PTO.
BR-3.3 · Must
System shall create an immutable snapshot of the credit memo at time of submission
Ops: Snapshot stored as versioned PDF + structured data. Retention: life of loan + 7 years. Tamper-evident hash.
BR-3.4 · Should
System shall enforce SLA timers with auto-escalation if approver does not act within threshold
Ops: SLA windows configurable per lane. Escalation chain: primary approver → backup → CCO. Weekly SLA compliance report.
BR-3.5 · Should
Banker shall see real-time deal status with push notifications on state changes
Ops: Push via FCM (Android) / APNs (iOS). Email fallback if push undeliverable. Status visible on deal card and portfolio dashboard.
4

Approve

Async review — approver acts from their own queue without blocking the lender's other work

C Customer Experience
Informed Promptly on Decision
Banker calls customer with approval/conditions. No more "I'll check on it" — banker has the answer in real time.
Conditions Are Clear
If conditional approval, customer gets a plain-language list of what's needed (appraisal, insurance, additional docs).
B Banker Workflow
Not Blocked — Works Other Deals
Lender continues building other deals while this one is in approval. No idle waiting.
Receives Decision Notification
Push notification with approve/conditional/decline + comments. Can view full approver notes immediately.
Address Conditions (if any)
Conditions appear as a checklist on the deal. Banker fulfills and checks off. Re-submit not needed for conditions — just documentation.
Approver
Approver Queue & Decision Workflow
Approver sees prioritized queue (oldest / highest-amount first). Full credit memo with risk summary, financial analysis, and prior relationship context. Three-button decision: Approve / Conditional Approve / Decline. All require written comments. Decision is audit-logged and immutable.
Operations Detail
  • Approver comments become part of permanent credit file — visible to examiners
  • Conditional approval generates a machine-readable condition list with due dates and responsible parties
  • Decline requires mandatory "decline reason" from enumerated list + free-text explanation
  • Committee approvals require quorum; system tracks votes and records dissents
  • Concentration limit checks run automatically before approval: borrower, industry, geography, collateral type
  • All approval actions produce an audit record: who, when, what, from which device/location
Business Requirements — Phase 4
Must Have (MVP) Should Have (v1) Could Have (v2)
BR-4.1 · Must
Approver shall have a prioritized queue showing pending deals with risk summary and key metrics
Ops: Queue sort: SLA urgency → amount → submission date. Mobile-optimized view for approvers on the go.
BR-4.2 · Must
System shall support Approve, Conditional Approve, and Decline decisions with mandatory comments
Ops: Decision record is immutable and exam-ready. Conditional items generate trackable checklist with due dates.
BR-4.3 · Must
All approval decisions shall be audit-logged with user, timestamp, device, and full decision rationale
Ops: Retention: life of loan + 7 years. Export-ready for examiner requests. SOX-compliant tamper evidence.
BR-4.4 · Should
System shall run concentration limit checks against portfolio thresholds before approval is recorded
Ops: Concentration limits: borrower, industry (NAICS), geography, collateral type. Thresholds set by ALCO. Breach = CCO escalation.
BR-4.5 · Could
System shall support committee voting workflow with quorum tracking and dissent recording
Ops: Quorum rules configurable. Dissent opinions recorded and retained. Committee minutes auto-generated from vote + comments.
5

Book & Close

Approved deal moves to loan ops for documentation, booking, and funding

C Customer Experience
Receives Commitment Letter
Clear terms, conditions, and timeline. Customer signs (wet or e-sig) and returns.
Provides Remaining Documents
Insurance, title work, environmental reports, appraisals. Banker tracks and nudges proactively.
Closing & Funding
Signs loan documents. Funds disbursed to their account or to closing agent. Deal is done.
B Banker Workflow
Generate Commitment Letter
System generates from approved deal terms. Banker reviews, adjusts language if needed, sends to customer.
Condition Tracking & Doc Checklist
Outstanding conditions and required documents shown as actionable checklist. Status visible to banker, loan ops, and approver.
Handoff to LOS / Loan Ops
Approved deal data pushed to loan origination system (nCino or target platform). Loan ops takes over doc prep and booking.
System
LOS Integration & Document Generation
Structured deal data pushed to LOS via API. Commitment letter, doc checklist, and closing package generated from templates. Dual-entry eliminated — data entered once in FUB Lender flows through to booking without re-keying.
Operations Detail
  • LOS integration is the primary v1/v2 dependency — MVP may use PDF export + manual entry as interim
  • Commitment letter templates maintained by legal; versioned and approved before production use
  • Condition fulfillment requires loan ops sign-off — banker cannot self-clear conditions
  • Funding authorization requires dual control (banker + loan ops) for amounts above threshold
  • Booking confirmation writes loan number back to FUB Lender deal record for lifecycle tracking
  • Exception tracking: any deal not closed within 90 days of approval triggers management review
Business Requirements — Phase 5
Must Have (MVP) Should Have (v1) Could Have (v2)
BR-5.1 · Must
System shall generate a printable/exportable credit memo PDF from approved deal data
Ops: PDF template approved by credit admin. Includes all fields, financials, conditions, and approval chain. Examiner-ready format.
BR-5.2 · Should
System shall generate commitment letter from approved terms using legal-approved templates
Ops: Templates version-controlled by legal. Custom language fields clearly marked. E-signature integration (v1).
BR-5.3 · Should
System shall push approved deal data to LOS via API, eliminating re-keying
Ops: API mapping document required per LOS. Data validation before push. Rollback if LOS rejects. MVP interim: PDF + manual entry.
BR-5.4 · Should
System shall track outstanding conditions and required documents with status and due dates
Ops: Conditions require loan ops sign-off. Auto-reminder at 7 days, 3 days, and 1 day before due date. Overdue escalation to manager.
BR-5.5 · Could
System shall support e-signature workflow for commitment letters and closing documents
Ops: Integration with DocuSign or equivalent. Wet-signature fallback always available. Signature audit trail retained.
6

Portfolio Management

Post-close lifecycle — banker maintains relationship, system monitors health

C Customer Experience
Proactive Relationship
Banker reaches out before maturity, before covenant deadlines — not after. Customer feels managed, not forgotten.
Renewal Is Seamless
When loan matures, banker already has updated financials and a pre-built renewal memo. Fast turnaround, minimal paperwork.
Cross-Sell Opportunities
Banker sees the full relationship — deposits, loans, treasury. Can identify and propose additional products naturally.
B Banker Workflow
Portfolio Dashboard
All active deals in one view. Sortable by maturity, risk rating, past-due status, covenant dates. Color-coded alerts.
Renewal & Review Pipeline
System surfaces upcoming maturities 90/60/30 days out. Pre-populates renewal memo from original deal + updated financials.
Exception & Covenant Monitoring
System tracks documentation exceptions, covenant compliance, and risk rating changes. Alerts on breaches or approaching thresholds.
Relationship Summary
Full borrower view: all loans, deposits, services, contact history. One screen before any customer call.
System
Portfolio Intelligence & Monitoring
Pulls loan performance data from core. Calculates portfolio-level metrics: weighted average risk rating, concentration percentages, maturity distribution, past-due trends. Feeds management reporting and board-level dashboards.
Operations Detail
  • Maturity alerts: 90-day (informational), 60-day (action required), 30-day (escalation if no renewal started)
  • Covenant tracking requires structured covenant entry at deal creation — not free-text
  • Risk rating migration tracked quarterly; downgrades trigger credit review workflow
  • Portfolio data refreshed daily from core via batch sync (real-time for payment/past-due status)
  • Management reports: portfolio concentration, past-due trend, new production, and pipeline — auto-generated monthly
  • Examiner reporting: criticized/classified loan list, ALLL analysis inputs, CRA data — exportable on demand
Business Requirements — Phase 6
Must Have (MVP) Should Have (v1) Could Have (v2)
BR-6.1 · Should
Banker shall have a portfolio dashboard showing all active deals with risk, maturity, and past-due indicators
Ops: Data from core via daily batch + real-time past-due feed. Dashboard loads <3s. Mobile-optimized.
BR-6.2 · Should
System shall surface upcoming maturities at 90/60/30 days and pre-populate renewal credit memos
Ops: Renewal memo pulls from original deal + most recent financials. Banker reviews and updates narrative. Same approval workflow applies.
BR-6.3 · Could
System shall track covenant compliance with automated breach detection and alerting
Ops: Requires structured covenant entry. Financial covenants auto-checked against spread data. Non-financial covenants require manual attestation.
BR-6.4 · Could
System shall generate management and examiner reports from portfolio data
Ops: Report templates approved by CFO/CCO. ALLL inputs, concentration reports, CRA data. Exportable as PDF/Excel.
