RECOMMENDATION

Expo with Custom Dev Client

This approach gives us the developer-experience advantages of Expo (EAS Build, OTA updates, managed upgrades) while unlocking the native modules we need for banking-grade security: WatermelonDB for offline storage, certificate pinning, jailbreak detection, and MDM/Intune integration.

The custom dev client means we compile our own development build (instead of using the Expo Go app), which lets us include native modules. In production, EAS Build handles iOS and Android compilation in the cloud — no local Xcode or Gradle setup required for Henry or future developers.

OTA updates via EAS allow us to push JavaScript-layer changes (UI, business logic, bug fixes) directly to devices without going through App Store review — critical for iterating on credit memo NLP accuracy and workflow adjustments post-launch.

Presentation Layer

React Native with Expo's custom dev client. Navigation uses React Navigation 7 with a bottom tab navigator (5 tabs) and stack navigators nested under each tab. All UI components come from a shared FUB Design System library that enforces brand colors, Lato typography, and the 50px pill-shaped CTA pattern.

Screen composition follows a "container/presenter" pattern: container components handle data fetching and state, presenter components are pure UI. This makes screens testable in isolation.

State Management: Zustand

Zustand was chosen over Redux for three reasons: minimal boilerplate (critical with a small team), built-in support for persistence middleware (pairs naturally with offline storage), and excellent TypeScript inference without extra configuration.

Global stores: AuthStore (user session, role, permissions), SyncStore (connectivity status, queue depth, last sync), UIStore (modals, toasts, navigation state).

Server State: TanStack Query

TanStack Query manages all server-state caching, refetching, and optimistic updates. Combined with WatermelonDB, it creates a two-tier cache: TanStack handles in-memory server state for fast renders, while WatermelonDB provides persistent offline storage that survives app restarts.

Stale-while-revalidate strategy: show cached data immediately, refresh in background. Query keys follow a convention: ['deals', dealId, 'memo'].

Network Layer: Axios + Queue

All API calls go through a centralized Axios instance with three middleware layers:

  1. Auth interceptor — attaches JWT, auto-refreshes expired tokens via Okta
  2. Retry middleware — exponential backoff (3 attempts, 1s/2s/4s) for 5xx and network errors
  3. Offline queue — when device is offline, mutations are serialized to a persistent queue and replayed on reconnection in order

Auth Flow: Okta OIDC with PKCE

Authentication uses the Authorization Code flow with PKCE (Proof Key for Code Exchange), which is the recommended pattern for mobile apps per OAuth 2.1. No client secret is stored on-device.

The flow: App opens Okta login in a system browser (ASWebAuthenticationSession on iOS, Chrome Custom Tab on Android) → user authenticates with FUB SSO credentials → Okta redirects back with auth code → app exchanges code for JWT tokens → access token (15 min) + refresh token (24 hr) stored in secure enclave.

Role-Based Access Control

Four roles, enforced server-side and reflected in the UI:

  • Banker — Create deals, write memos, submit for approval, view own pipeline
  • Approver — All Banker permissions + approve/decline within authority limit
  • Loan Admin — All permissions + book/close deals, manage document checklist
  • Read-Only — View pipeline and deal details, no mutations. For auditors and executives

Offline-First Architecture

Many FUB community banks are in rural Oklahoma and Texas where cellular coverage is inconsistent. A banker at a cattle ranch in Pauls Valley can't lose their work because they lost signal. This is the most critical architectural decision in the entire app.

The Jackie Parsons Test

Our canonical scenario: Banker drives to Jackie's ranch outside Pauls Valley, OK. Opens the app on the drive while still in town (has signal). Arrives at the ranch — no signal. Opens a new deal, writes a credit memo for Jackie's $110K cattle operating line (50 head Black Angus, 7-year customer, perfect payment history). Takes photos of financial documents. Saves everything. Drives back to town — app syncs automatically. The deal shows up in the approver's queue within 30 seconds of reconnection.

If this scenario doesn't work flawlessly, we haven't built the right thing.

Data Layer: WatermelonDB

WatermelonDB sits on top of SQLite and provides a reactive, observable database optimized for React Native. It was chosen over raw SQLite, Realm, or MMKV because it was purpose-built for offline-first apps with sync, has lazy record loading (critical for large pipelines), and uses an observable pattern that pairs naturally with React's rendering model.

Conflict Resolution: Last-Write-Wins + Merge Log

When the same record has been modified both locally and on the server, we use a last-write-wins strategy at the field level (not the record level). This means if Banker A updates the loan amount while offline, and Banker B updates the collateral description on the server, both changes are preserved.

Every conflict produces a merge log entry visible in the deal's activity feed, so nothing happens silently. If a true collision occurs (same field, different values), the most recent timestamp wins and the overwritten value is logged for review.

Encryption at Rest

The local SQLite database is encrypted using SQLCipher. Encryption keys are stored in the platform's secure enclave — iOS Keychain (with kSecAttrAccessibleAfterFirstUnlock) and Android Keystore. The key never exists in user-accessible storage or application memory beyond the decryption operation.

If the device is jailbroken/rooted, the app refuses to decrypt the local database and requires re-authentication over network.

Sync Priority Order

When the device reconnects, queued operations sync in this order to maximize business-critical throughput:

  1. Approval actions — highest priority, a delayed approval blocks the whole deal pipeline
  2. Deal mutations — new deals, memo updates, status changes
  3. Document uploads — largest payloads, streamed with resumable upload support
  4. Read-state sync — pipeline refreshes, notification acknowledgments

Offline Window: 72 Hours

The app supports up to 72 hours of offline operation. After 72 hours without a successful sync, the app enters a "stale data" warning state — the banker can still view and edit, but a prominent banner warns that data may be out of date.

At 7 days without sync, the app requires re-authentication before allowing new mutations (but existing queued changes are preserved and will sync once authenticated). This protects against lost/stolen devices operating indefinitely with cached data.

Deep Linking

The app supports deep links for push notification routing. When an approver taps a "New deal pending approval" notification, the app navigates directly to fublender://approvals/:dealId. URL scheme: fublender:// for development, https://lender.fub.com for universal links in production.

API Contract Summary

All endpoints go through the MuleSoft API gateway. Base URL: https://api.fub.com/lender/v1. Authentication: Bearer JWT from Okta. Rate limiting: 100 req/min per user.

Deals

GET /api/deals List deals with filters

Query params: ?lane=fast|standard|complex &status=draft|submitted|approved|booked &banker_id= &search= &page=&limit=

Response: Paginated array of deal summaries (id, borrower name, amount, lane, status, last_updated, days_in_status).

Scope: Bankers see own deals. Approvers see deals in their queue + own. Loan Admin sees all. Read-Only sees all.

POST /api/deals Create new deal

Body: { borrower_id, loan_type, requested_amount, purpose, lane (auto-detected or manual override) }

Lane auto-detection: If requested_amount < 250000 and loan_type is not SBA/participation → Fast Lane. Otherwise Standard or Complex based on entity count and loan type.

Returns: Full deal object with generated deal_id and initial status draft.

GET /api/deals/:id Full deal detail

Returns: Complete deal object including borrower info, memo summary, document list, approval chain, status history, conditions, and linked entities.

Credit Memo

GET /api/deals/:id/memo Get credit memo

Returns: Memo narrative text, NLP-extracted structured fields (if processed), extraction confidence scores, last editor, version history.

PUT /api/deals/:id/memo Update memo + trigger NLP

Body: { narrative_text, manual_overrides? }

Behavior: Saves the raw narrative text. Triggers async NLP extraction pipeline. The extraction runs server-side and populates structured fields (loan amount, purpose, collateral, financial ratios, risk factors). Banker can manually override any extracted field.

NLP fields extracted: requested_amount, loan_purpose, collateral_description, collateral_value, borrower_years_with_bank, payment_history_summary, financial_highlights, risk_factors, mitigants, recommendation.

Approvals

POST /api/deals/:id/approve Approval action

Body: { action: "approve" | "decline" | "return_for_info" | "escalate", conditions?, comments?, authority_code }

Server validation: Checks that the approver has sufficient authority for the deal amount. Validates dual control (cannot approve own deal). Logs action with timestamp, device fingerprint, and approximate location. If amount exceeds approver's limit, auto-escalates to next authority level.

Returns: Updated deal with new status, next approver (if escalated), and complete approval chain.

Documents

POST /api/deals/:id/documents Upload document

Body: Multipart form — file binary + metadata { document_type, description, source: "camera" | "upload" | "email" }

Server-side: File stored in OpenText UView. OCR runs asynchronously. Auto-classification attempts to categorize (financial statement, tax return, insurance certificate, etc.). Results returned via webhook or polling.

Resumable uploads: Uses tus protocol for large files. Upload can resume after connectivity loss.

Core Banking Integration

GET /api/core/borrower/:id Jack Henry borrower lookup

Route: App → MuleSoft → JHA Jxchange → SilverLake (on-premise)

Returns: Borrower demographics, existing loan portfolio, deposit relationships, payment history, risk rating, officer assignment, TIN/SSN (masked, last 4 only in app).

Caching: Borrower data cached locally for 24 hours for offline access. Fresh lookup always attempted when online.

Latency note: SilverLake on-premise queries can take 2-5 seconds via Jxchange. The app shows a skeleton loader and pre-populates from cache while the fresh lookup completes.

GET /api/core/borrower/search Search borrowers in core

Query params: ?q= (name, TIN last 4, account number) &limit=20

Returns: Array of borrower summaries for selection. Cached results available offline for previously-searched borrowers.

Authentication

POST /api/auth/token Exchange auth code for JWT

Body: { grant_type: "authorization_code", code, code_verifier, redirect_uri }

Returns: { access_token (15 min), refresh_token (24 hr), id_token, user_profile { role, authority_level, community_bank, full_name } }

POST /api/auth/refresh Refresh expired access token

Body: { grant_type: "refresh_token", refresh_token }

Returns: New access_token + refresh_token pair. Old refresh token is invalidated (rotation).

Authority Enforcement

Lending authority is enforced server-side. The app never makes an authority decision — it only renders what the server allows. If the server says "over your limit," the app shows the escalation path. Period.

Iron Rule: No Client-Side Authority Logic

The React Native app contains zero authority validation code. It does not know authority limits. It does not calculate whether an approver can approve. All authority checks happen on the server behind the /api/deals/:id/approve endpoint. The app simply sends the approval action and handles the server's response — which is either "approved" or "escalated to [next approver]" or "insufficient authority."

This is non-negotiable. Authority logic in client code is a regulatory audit failure waiting to happen.

Enforcement Rules

Dual Control

No person can approve a deal they originated. The server rejects any approval where approver_id === originating_banker_id. This is checked at the API level, not in the app UI. The app hides the "Approve" button for your own deals as a UX convenience, but the server is the enforcement mechanism.

Auto-Escalation

When a deal's total exposure exceeds the approver's individual limit, the server automatically routes to the next authority level. The approver can still add conditions and comments, but their action is recorded as "recommended for approval" rather than "approved." The deal moves up the chain until it reaches someone with sufficient authority.

Audit Trail

Every approval action is logged immutably with: timestamp (server clock, not device clock), approver identity (from JWT, not self-reported), device fingerprint (device ID + OS version), approximate location (if permissions granted), action taken, conditions attached, and the deal snapshot at time of approval.

Concentration Limits

The server also checks aggregate exposure: total loans to a single borrower, industry concentration, and geographic concentration. These checks run at approval time and can trigger additional routing to Credit Risk even if the individual deal is within the approver's limit.

Security Architecture

Banking-grade security for a mobile app carrying borrower PII, financial data, and lending decisions. Defense in depth — every layer assumes the layer above it has been compromised.

Certificate Pinning

All API calls to api.fub.com use certificate pinning via a custom Expo config plugin. The app ships with a pinned certificate hash and rejects any TLS connection that doesn't match. This prevents MITM attacks even on compromised Wi-Fi networks (common at conference venues and hotels where bankers travel).

Pin rotation strategy: the app pins to the intermediate CA certificate (not the leaf), and includes one backup pin. OTA updates can push new pins without an app store release.

Biometric + PIN Access

App launch requires biometric authentication (FaceID on iPhone, fingerprint on Android) or a 6-digit PIN fallback. Biometric is tied to the device's secure enclave — we use expo-local-authentication with requireConfirmation: true.

After 5 failed PIN attempts, the app locks and requires full Okta re-authentication. Biometric enrollment is mandatory at first login; PIN is a secondary fallback for cases where biometric hardware fails.

Session Management

Access tokens expire after 15 minutes. Refresh tokens last 24 hours with rotation (each use generates a new refresh token and invalidates the old one). After 15 minutes of inactivity (no screen touches), the app requires biometric re-authentication but preserves the session — the user doesn't lose unsaved work.

Background timeout: if the app is backgrounded for more than 5 minutes, it requires biometric on return. Screen capture is disabled on iOS (UIScreen.main.isCaptured check) and Android (FLAG_SECURE).

Jailbreak / Root Detection

On launch, the app checks for jailbreak (iOS) or root (Android) indicators using a native module. Checks include: presence of Cydia/Sileo, su binary detection, writable system partitions, Magisk markers, and Frida detection. If detected, the app refuses to launch and displays a message directing the user to contact IT.

These checks run as native code (not JavaScript) to resist runtime patching. The server also validates a device attestation token (Apple DeviceCheck / Google Play Integrity) at each login.

Data Protection

No sensitive data in logs: all logging uses a sanitizer that strips PII fields (SSN, TIN, account numbers, names) before writing to crash reporting (Sentry). Log level in production: warn only — no debug or info logs ship.

Clipboard protection: sensitive fields (SSN, account numbers) are marked non-copyable in the UI. The app clears the clipboard if it detects the app entering background with sensitive data copied.

MDM / Intune Enrollment

The app is distributed via Microsoft Intune (MAM without enrollment for BYOD, full MDM for bank-owned devices). Intune App Protection Policies enforce: minimum OS version, encryption requirement, and prevent data transfer to unmanaged apps.

The app checks for Intune management status at launch. On unmanaged devices, the app operates in a restricted mode where document downloads are disabled and all data is encrypted with app-level keys that Intune can remotely wipe.

Integration Architecture

The app is a thin client. All integrations go through the MuleSoft API gateway — the app never talks directly to downstream systems.

Integration Architecture — All downstream system access is mediated through MuleSoft

Jack Henry SilverLake (via Jxchange)

SilverLake is on-premise. The JHA Jxchange cloud API layer provides RESTful access to core banking data. Latency is 2-5 seconds per call due to the on-prem bridge. MuleSoft handles connection pooling, retry logic, and response caching (5-minute TTL for borrower demographics, no caching for balance/transaction data).

Key operations: Borrower lookup, loan booking (post-approval), payment history pull, risk rating retrieval, deposit relationship summary.

nCino / Salesforce (Read-Only Coexistence)

nCino remains the system of record for regulatory reporting and existing loan portfolio management. FUB Lender writes deal data to nCino via the Salesforce API after deals are approved, creating mirror records. Bankers never need to touch nCino for new originations — the sync is invisible to them.

Sync timing: Near-real-time (within 5 minutes of approval). If sync fails, the deal is flagged in a reconciliation queue for Loan Admin review.

OpenText UView

All deal documents are stored in OpenText, which is FUB's enterprise document management system. The app uploads documents via MuleSoft, and OpenText handles versioning, retention policies, and regulatory holds. OCR and auto-classification run as OpenText workflows.

DocuSign

E-signatures for loan documents, commitment letters, and compliance disclosures. The app creates DocuSign envelopes via API, with signing order determined by the deal type (borrower signs first for consumer, bank officer signs first for commercial commitments). Status webhooks update the deal's document checklist in real-time.

Decisions Log

Five decisions that need stakeholder alignment before Henry writes the first line of code. Each has a recommendation, but these require explicit sign-off from Justin, Ryan, and (where noted) IT Security.

Decision 1

Expo Custom Dev Client vs. Bare React Native

Recommendation: Expo with custom dev client. This gives us EAS Build (cloud compilation, no local Xcode/Gradle), OTA updates for rapid iteration, and managed RN upgrades — while still supporting the native modules we need (WatermelonDB, cert pinning, jailbreak detection). The tradeoff is a slight abstraction layer over native code, but this is well worth it given our team size (Henry as primary developer).

What changes if rejected: Bare RN means Henry needs Xcode and Android Studio locally, manual RN version upgrades (painful), and we lose OTA updates unless we set up CodePush. Development velocity drops approximately 30%.

Sign-off needed: Justin Sanderson, Henry Marquez
Decision 2

WatermelonDB for Offline Storage vs. Alternatives

Recommendation: WatermelonDB over Realm, raw SQLite, or MMKV. WatermelonDB was purpose-built for offline-first React Native apps with sync capabilities. It uses lazy record loading (critical when a banker's pipeline has 200+ deals), has a built-in sync protocol, and its observable pattern works naturally with React's rendering model. Realm was considered but adds a heavier dependency and has had React Native compatibility issues with new architectures.

What changes if rejected: Raw SQLite means writing our own sync engine from scratch (3-4 weeks of additional development). Realm is viable but introduces a MongoDB dependency and has licensing considerations for commercial use.

Sign-off needed: Henry Marquez, Justin Sanderson
Decision 3

72-Hour Offline Window

Recommendation: Allow 72 hours of offline operation before requiring re-authentication. This covers the worst-case rural scenario: banker visits a remote ranch on Friday afternoon, loses signal, doesn't return to town until Monday morning. The 72-hour window means no data loss. After 72 hours, the app warns about stale data but doesn't block usage. After 7 days, it requires online re-auth.

What changes if rejected: A shorter window (e.g., 24 hours) reduces stale data risk but means bankers in very rural areas could lose access during multi-day stretches. A longer window (e.g., 7 days) increases the risk of conflicting edits on deals being worked by multiple parties. IT Security may have a position on maximum offline duration for devices carrying borrower PII.

Sign-off needed: Ryan Suchala, IT Security, Justin Sanderson
Decision 4

nCino Coexistence Strategy: Read-Only Mirror

Recommendation: Write approved deals to nCino as read-only mirror records via Salesforce API. This maintains regulatory reporting continuity and lets Credit Risk and Loan Review teams continue using nCino's portfolio analytics while bankers originate in the new app. nCino becomes a downstream consumer, not an input system.

What changes if rejected: If we don't sync to nCino, the regulatory reporting team loses visibility until we build equivalent reporting in the new system (6+ months). If we make it bidirectional, we introduce complexity that undermines the purpose of building a new app. The recommendation is one-way sync: FUB Lender → nCino, never the reverse.

Sign-off needed: Ryan Suchala, Chief Risk Officer, Compliance, Justin Sanderson
Decision 5

MDM Strategy: Intune MAM (BYOD) + Full MDM (Bank Devices)

Recommendation: Dual-mode deployment. Bank-owned devices get full MDM enrollment via Intune, giving IT complete control (remote wipe, policy enforcement, app restrictions). Banker-owned (BYOD) devices use Intune Mobile Application Management (MAM) without device enrollment — the app is containerized with its own encryption, but the bank doesn't control the entire device. This respects banker privacy on personal devices while protecting bank data.

What changes if rejected: Requiring full MDM for all devices (including BYOD) will face resistance from bankers who don't want IT managing their personal phones. Going MAM-only for all devices reduces IT's ability to remotely manage bank-owned hardware. The hybrid approach is industry standard for community banks.

Sign-off needed: IT Security, CIO, Ryan Suchala, Justin Sanderson
