Key Schema Design Decisions

TIN hashing, not encryption

TINs are stored as SHA-256 hashes. The app never needs to display a full TIN — it only needs to verify uniqueness and match against Jack Henry records. This eliminates an entire class of data-at-rest exposure. The raw TIN lives only in Jack Henry SilverLake.

Append-only audit table

The audit_events table has UPDATE and DELETE revoked at the PostgreSQL role level. The application database user physically cannot modify history. A separate read-only replica feeds the compliance dashboard. Examiner-grade immutability.

JSONB for NLP extraction

The nlp_extracted_fields column in credit_memos stores structured output from NLP without requiring schema migrations as the extraction model improves. GIN indexes on JSONB paths support fast queries on specific extracted fields (e.g., DSCR, collateral type).

Conflict-aware sync queue

Offline edits from field bankers queue in sync_queue with full payloads. On reconnect, the server applies last-write-wins for non-critical fields and flags conflicts for manual resolution on approval actions — a banker can't unknowingly overwrite an approver's decision.

API Endpoint Design

RESTful API served via MuleSoft API Gateway → Azure AKS (Node.js / NestJS). All endpoints require Okta JWT bearer token. Rate limits: 100 req/min per user, 1000 req/min per service account. API versioning via URL path (/api/v1/).

Server-Side Middleware — Authority Validation

authority.middleware.ts — NestJS GuardTypeScript
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';

@Injectable()
export class AuthorityGuard implements CanActivate {
  async canActivate(context: ExecutionContext): Promise<boolean> {
    const req = context.switchToHttp().getRequest();
    const deal = await this.dealService.findById(req.params.id);
    const user = req.user; // from Okta JWT

    // Rule 1: Self-approval prevention
    if (deal.created_by === user.id) {
      throw new ForbiddenException(
        'SELF_APPROVAL_BLOCKED',
        'Originator cannot approve their own deal'
      );
    }

    // Rule 2: Authority level check
    const requiredLevel = await this.getRequiredLevel(deal.amount);
    const userLevel = await this.getUserAuthorityLevel(user.id);

    if (userLevel.level_number < requiredLevel) {
      throw new ForbiddenException(
        'INSUFFICIENT_AUTHORITY',
        `Requires Level ${requiredLevel}, user has Level ${userLevel.level_number}`
      );
    }

    // Rule 3: Relationship exposure check
    const totalExposure = await this.calcRelationshipExposure(
      deal.borrower_id, user.id
    );
    if (totalExposure + deal.amount > userLevel.relationship_limit) {
      throw new ForbiddenException(
        'RELATIONSHIP_LIMIT_EXCEEDED',
        `Total exposure $${totalExposure + deal.amount} exceeds limit`
      );
    }

    // Rule 4: Committee requirement
    if (deal.amount > 1000000 || deal.requiresCommittee) {
      if (!user.roles.includes('committee_member')) {
        throw new ForbiddenException(
          'COMMITTEE_REQUIRED',
          'Deals over $1M require Loan Committee approval'
        );
      }
    }

    return true;
  }

  // Determine minimum authority level for a given amount
  private async getRequiredLevel(amount: number): Promise<number> {
    const level = await this.authorityRepo
      .createQueryBuilder('am')
      .where('am.individual_limit >= :amount', { amount })
      .andWhere('am.effective_date <= CURRENT_DATE')
      .orderBy('am.level_number', 'ASC')
      .getOne();
    return level?.level_number ?? 6; // default to committee
  }
}

Escalation Routing Algorithm

When a deal is submitted, the routing service finds the next available approver who (a) has sufficient authority, (b) is not the originator, and (c) is within the originator's market or region. If no one is available within the SLA window, it escalates one level up.

routing.service.ts — Approver selectionTypeScript
async routeDealForApproval(deal: Deal): Promise<RoutingResult> {
  const requiredLevel = await this.getRequiredLevel(deal.amount);
  const slaHours = this.getSlaByLane(deal.lane);
  // { fast: 4, standard: 24, complex: 72 }

  // Step 1: Find approvers at the required level in same market
  let candidates = await this.findApprovers({
    minLevel: requiredLevel,
    marketId: deal.originMarketId,
    excludeUserId: deal.created_by,  // never self-approve
    isAvailable: true,               // not OOO
  });

  // Step 2: If no local approvers, widen to region
  if (candidates.length === 0) {
    candidates = await this.findApprovers({
      minLevel: requiredLevel,
      regionId: deal.originRegionId,
      excludeUserId: deal.created_by,
      isAvailable: true,
    });
  }

  // Step 3: If still empty, escalate to next level
  if (candidates.length === 0) {
    return this.routeDealForApproval({
      ...deal,
      amount: deal.amount, // triggers higher level match
      _escalated: true,
    });
  }

  // Step 4: Select approver with lightest current queue
  const selected = candidates.sort(
    (a, b) => a.pendingDealCount - b.pendingDealCount
  )[0];

  // Step 5: Set SLA timer
  await this.slaService.startTimer({
    dealId: deal.deal_id,
    approverId: selected.id,
    deadlineHours: slaHours,
    escalationAction: 'notify_next_level',
  });

  return { approverId: selected.id, slaDeadline: slaHours };
}

Authentication Architecture

Identity Layer — Okta

All users authenticate via Okta with SAML 2.0 for web and OIDC/PKCE for mobile. JWT access tokens contain: sub (user ID), authority_level, market_id, region_id, roles[]. Tokens expire in 15 minutes; refresh tokens last 8 hours. Biometric unlock (Face ID / fingerprint) re-issues tokens from a secure device-bound refresh token — no stored passwords on device.

Approval MFA Step-Up

Any approval action (approve, decline, condition) triggers an Okta step-up challenge. The approver must re-authenticate with biometrics or Okta Verify push notification. The step-up token is single-use and bound to the specific deal_id + action pair. This prevents replay attacks where a captured approval token is reused on a different deal.

Dual Control for High-Value

Deals above $500K require two independent approvals (different users, different authority levels). The first approval moves the deal to in_review with a flag; the second moves it to approved. Both approvers' device fingerprints and locations are logged.

Offline Token Handling

When offline, the mobile app can read cached deal data but cannot approve. Draft memo edits are queued in sync_queue with a signed device attestation. On reconnect, the sync service validates each queued operation against current auth state — a revoked user's offline edits are rejected.

Threat Model — Top 10

STRIDE-based threat analysis focused on the lending approval pipeline. Each threat includes attack vector, impact assessment, and layered mitigations. Ordered by risk score (likelihood × impact).

T-01
Critical
Insider Self-Approval Bypass
A banker with approver credentials approves their own deal, either directly or by manipulating the assigned_approver field via API tampering.
Mitigations Server-side guard compares deal.created_by against req.user.id — no client-side check dependency. The assigned_approver field is set only by the routing service, never accepted from client input. Audit log flags any approval where originator and approver share the same market, triggering compliance review.
T-02
Critical
Stolen Device with Cached Credentials
A lost or stolen phone with an active session and cached deal data. Attacker attempts to approve deals using biometric bypass (e.g., printed fingerprint).
Mitigations Approval actions require Okta step-up MFA (push or biometric), not just session token. Device attestation validates iOS Secure Enclave / Android Keystore integrity. Remote wipe via Okta device management revokes all tokens. Offline mode is read-only — no approval actions can be queued offline.
T-03
High
SIM Swap on Approver's Phone
Attacker performs SIM swap to intercept SMS-based MFA, then approves deals from a different device.
Mitigations No SMS-based MFA — Okta Verify push and biometric only. Device fingerprint is logged with every approval; a new device triggers a mandatory re-enrollment flow that requires in-person identity verification. Anomaly detection flags approvals from unrecognized devices.
T-04
High
SQL Injection on Search/Filter
Malicious input in the borrower search or deal list filter parameters targets the PostgreSQL backend.
Mitigations All database access uses parameterized queries via TypeORM query builder — no string concatenation. MuleSoft API Gateway applies input validation regex before requests reach AKS. WAF rules on Azure Front Door block common injection patterns. Quarterly penetration testing with reported findings.
T-05
Critical
Document Tampering Post-Approval
After a deal is approved, someone replaces a financial statement in OpenText with a falsified version to cover fraud.
Mitigations SHA-256 hash of every document is stored in the documents table at upload time. A nightly integrity job re-hashes documents in OpenText and compares. Post-approval documents are locked in OpenText with read-only retention policy. Any hash mismatch triggers an immediate alert to the CRO and freezes the deal.
T-06
High
Privilege Escalation (Banker → Approver)
A banker manipulates their Okta group membership or JWT claims to gain approver-level authority they shouldn't have.
Mitigations Authority level is stored in both Okta (source of truth) and the authority_matrix table. The middleware cross-references both — a mismatch logs an alert and denies the action. Okta group changes require admin approval and trigger an audit event. JWTs are validated against Okta's JWKS endpoint on every request, not just at login.
T-07
High
Offline Queue Poisoning
An attacker with physical device access injects malicious payloads into the sync queue while the device is offline, hoping they'll be applied when connectivity returns.
Mitigations Each sync queue entry includes a device-signed attestation token. The sync service validates the signature against the device's registered public key. Payloads are schema-validated server-side before applying. Approval actions are never accepted from the sync queue — they require live connectivity and step-up MFA.
T-08
Medium
API Key / Token Theft
Service account credentials for the MuleSoft → AKS connection or the Jxchange API are compromised, allowing unauthorized access to borrower data or deal creation.
Mitigations Service-to-service auth uses Azure Managed Identities (no static keys). MuleSoft API keys are rotated every 90 days and stored in Azure Key Vault. All service accounts have narrowly scoped permissions — the Jxchange service account is read-only. Network policies restrict AKS pod communication to whitelisted services only.
T-09
High
OFAC Screening Bypass
A deal is approved for a borrower who appears on the OFAC SDN list, either because screening was skipped or the borrower's name was slightly altered to evade matching.
Mitigations OFAC screening is a mandatory step in the submit workflow — the API rejects submission if screening hasn't completed. Fuzzy matching with a configurable threshold (default 85%) catches name variations. Screening results are stored as audit events and cannot be overridden without a compliance officer's approval. Daily re-screening of all active borrowers against updated OFAC lists.
T-10
Critical
Audit Log Tampering
An insider with database access modifies or deletes audit records to cover fraudulent approvals.
Mitigations UPDATE and DELETE are revoked on audit_events at the PostgreSQL role level — the application user physically cannot modify records. A separate read-only replica streams audit data to Azure Log Analytics (immutable retention). Database superuser access requires dual approval and is logged via Azure PIM. Quarterly reconciliation compares primary audit table row counts against the replica.

WF-05: Jack Henry Core Sync (Post-Booking Write-Back)

Writes approved and booked deals back to SilverLake via Jxchange

This is the only write operation to Jack Henry. All other core interactions are read-only lookups. The workflow includes retry logic with exponential backoff (max 3 retries over 15 minutes). On permanent failure, the deal is flagged as booking_failed and the ops team is notified for manual core entry.

WF-06: Nightly Portfolio Health Scoring

Calculates risk scores across the active portfolio
Cron: 2:00 AM CT daily
Query all deals WHERE status IN (booked, closed) → Pull latest financials from Jxchange → Calculate current DSCR, LTV, risk metrics → Flag deteriorating credits (DSCR < 1.1 or LTV > 85%) → Generate portfolio_health_scores table → Email daily digest to CRO

Scoring model weights: payment history (30%), current DSCR (25%), LTV ratio (20%), collateral volatility (15%), relationship tenure (10%). Credits that drop below watch thresholds are auto-flagged and appear on the CRO's morning dashboard.

nCino Coexistence Layer

FUB Lender runs alongside nCino during the transition period. The coexistence layer provides read-only access to nCino data via the Salesforce REST API, enabling bankers to reference historical deals without switching systems. No data flows from FUB Lender into nCino.

Risk context: nCino carries a three-layer dependency stack (nCino → Salesforce → AWS). This coexistence design deliberately avoids adding a fourth dependency layer. FUB Lender reads from nCino but never depends on it for any write operation or approval flow. If nCino goes down, FUB Lender continues to operate independently.

Salesforce API Connection

Authentication

OAuth 2.0 JWT Bearer flow with a Salesforce Connected App. The private key is stored in Azure Key Vault. Token refresh is handled by MuleSoft's built-in OAuth provider policy. No user credentials are stored or proxied — this is a system-to-system integration.

Rate Limits & Caching

Salesforce API calls are capped at FUB's org limit. To stay well under, all nCino reads are cached in Redis with a 30-minute TTL. Bankers see a "last synced" timestamp on nCino data. Cache is invalidated on-demand if a banker explicitly refreshes.

Migration Path — Sunsetting nCino

The coexistence layer is designed to be turned off without any schema changes to FUB Lender. The migration follows three phases.

PhaseDurationActionnCino State
Phase 1 Months 1–6 Parallel operation. New deals created in FUB Lender. Existing deals remain in nCino. Bankers can view nCino history from within FUB Lender. Active (read/write for legacy)
Phase 2 Months 6–12 nCino goes read-only. All new origination, approval, and booking happens in FUB Lender. Historical nCino data is batch-migrated to PostgreSQL for long-term access. MuleSoft continues to serve cached nCino data. Read-only
Phase 3 Month 12+ nCino Salesforce org is decommissioned. All historical data lives in PostgreSQL. The coexistence API endpoints return data from the migrated tables instead of Salesforce. MuleSoft nCino policies are removed. Decommissioned
Key safeguard: The batch migration in Phase 2 includes a reconciliation step that compares nCino record counts and checksums against the migrated PostgreSQL data. Migration isn't marked complete until 100% parity is verified and signed off by the CRO.
